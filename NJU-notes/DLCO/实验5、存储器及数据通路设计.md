# 5.1 只读存储器实验
## 整体方案设计

#### 1. 顶层设计模块图描述
本实验的顶层模块逻辑架构可以概括为 **“输入校验 -> 地址变换 -> 并行检索 -> 点阵驱动”**。

*   **设计思想**：由于字库文件是线性的（每个字符占 16 字节），而 LED 点阵需要同时接收 16 行数据。我们采用**存储器阵列** 的思想，将 16 个加载了相同字库内容的 ROM 并联。通过给每个 ROM 施加一个固定的地址偏移量（0~15），实现单次寻址即可同步取出该字符的完整 16 字节点阵。

#### 2. 子模块详细描述
1.  **范围检查器**：
    *   利用比较器判断输入的 ASCII 码。若码值 $< 0x20$ 或 $> 0x7F$，则判定为不可见字符。
2.  **地址计算单元**：
    *   **减法器**：执行 $Base = Code - 0x20$，将 ASCII 码映射到从 0 开始的字符索引。
    *   **左移器**：将差值左移 4 位（相当于 $\times 16$），得到该字符在字库中的绝对起始地址。
3.  **ROM 存储阵列**：
    *   由 16 个 11 位地址、8 位数据的只读存储器组成。
    *   所有 ROM 加载相同的 `ascii8-16.zk` 镜像。
4.  **LED 驱动模块**：
    *   将 16 个 ROM 的 8 位输出分别对应连接至 LED 点阵的 16 行输入端。

#### 3. 引脚定义与传输通道
*   **输入引脚**：`Code` (7-bit ASCII 码)。
*   **输出引脚**：`row1 ~ row16` (8-bit 数据线)，`CodeErr` (1-bit 错误标志)。
*   **传输通道**：
    *   **地址总线**：11 位，由计算单元产生的起始地址经过加法器链（$+0, +1, \dots, +15$）分发至各 ROM。
    *   **数据总线**：16 条 8 位独立通道，将字形信息并行传给点阵。

# 5.2数据存储器实验
## 整体方案设计
#### 1. 顶层设计模块图描述
本实验的顶层模块名为 `RAM_Module`。设计采用了 **“分块存储+ 路由选通”** 的架构。

*   **设计思想**：将 256KB 空间拆分为 4 个独立的 8 位存储体（$RAM_3 \sim RAM_0$）。这种设计允许我们通过控制每一片的片选信号（$SEL_i$），实现对任意字节、半字或整字的精准存取，而无需复杂的位移运算。
*   **层次结构**：
    1.  **地址与对齐检查逻辑**：分析 $MemOp$ 和地址低 2 位，生成片选信号及对齐错误标志。
    2.  **写数据路由逻辑**：将 32 位输入 $Din$ 的对应字节“路由”到目标 RAM 的输入端。
    3.  **存储阵列**：4 片 $64K \times 8$-bit RAM，通过地址线 $Addr[17:2]$ 并行寻址。
    4.  **读数据选通与扩展逻辑**：从 4 片 RAM 中提取目标数据，并根据 $MemOp$ 进行 0 扩展或符号扩展，最终拼装为 32 位。

#### 2. 子模块详细描述
1.  **控制信号译码器 (Control Decoder)**：
    *   输入 $MemOp[2:0]$ 和 $Addr[1:0]$。依据表 5.2 逻辑，输出 $SEL_{3:0}$。若访问不满足自然对齐（如地址为 1 时读取字），则令 $AlignErr=1$。
2.  **写通路多路开关**：
    *   根据 $Addr[1:0]$，决定 $Din$ 的低字节是给 $RAM_0, RAM_1, RAM_2$ 还是 $RAM_3$。
3.  **读通路选择与扩展器 (Read Post-Processor)**：
    *   利用多路选择器提取目标字节/半字。
    *   **符号扩展单元**：根据 $MemOp[2]$ 判断，若为有符号加载（`lb`, `lh`），则将提取数据的最高位复制到高 24/16 位。

#### 3. 引脚定义与传输通道
*   **主要输入**：`Addr[17:0]`（18位地址）、`Din[31:0]`、`MemOp[2:0]`、`WE`（写使能）。
*   **主要输出**：`Dout[31:0]`、`AlignErr`（对齐错误）。
*   **数据通路**：$Din \xrightarrow{分线器} \text{写选择逻辑} \xrightarrow{8-bit} RAM[0:3] \xrightarrow{8-bit} \text{读选通逻辑} \xrightarrow{位扩展} Dout$。

# 5.3 取指令部件实验 
## 整体方案设计
取指令部件（Instruction Fetch Unit, IFU）是单周期 CPU 数据通路的最前端，负责在每个时钟周期的上升沿根据当前程序计数器（PC）的值，从指令存储器（Instruction ROM）中取出对应的 32 位 RISC-V 指令，并根据指令类型与跳转控制逻辑实时计算出下一条指令的地址。

---

## 一、 顶层数据通路与控制逻辑设计框图

IFU 的顶层结构由**程序计数器（PC）**、**指令存储器（Instruction ROM）**、**地址加法计算链**以及**多路选择器（MUX）**构成。

```text
                     +---------------------------------------+
                     |             下地址逻辑                |
                     |   (Next PC Generation Logic)          |
                     |                                       |
                     |     +-----+                           |
 BusA (Rs1) -------> | 1 ->|     |                           |
                     |     | MUX |-+                         |
  PC (Current) ----> | 0 ->|     | | (NPCASrc)               |
                     |     +-----+ |                         |
                     |             v                         |
                     |          +-----+                      |
                     |          | Add | ----+ (Next PC)      |
                     |          +-----+     |                |
                     |             ^        |                |
                     |     +-----+ |        |                |
   Imm ------------> | 1 ->|     | |        |                |
                     |     | MUX |-+        |                |
     4 ------------> | 0 ->|     |   (NPCBSrc)               |
                     |     +-----+          |                |
                     +----------------------|----------------+
                                            |
   Reset (高电平有效)                       |
     |                                      |
     v                                      v
  +-------------------------------------------+
  |               PC 寄存器 (32-bit)          |
  |  - 若 Reset=1: PC <- Initial Address      |
  |  - 若 Reset=0: PC <- Next PC (CLK上升沿)   |
  +-------------------------------------------+
     |
     +-------> PC 物理地址输出 (32-bit)
     |
     v [17:2]
  +-------------------------------------------+
  |       指令存储器 ROM (256KB, 16位地址)    | --> IR 指令输出 (32-bit)
  +-------------------------------------------+
```

---

## 二、 子模块详细描述与技术实现

取指令部件（IFU）内部由三个核心功能子模块级联构成：

### 1. 程序计数器模块 (Program Counter, PC)
*   **功能描述**：用于保存当前正在执行指令的 32 位物理内存地址。在系统复位信号（Reset）有效时，PC 异步或同步强制初始化为指定的程序入口首地址（`Initial Address`）；在正常运行状态下，每个时钟信号（CLK）的上升沿到来时，采样并锁存下一条指令的计算地址（`Next PC`）。
*   **时序特征**：上升沿触发，具备高电平同步复位功能。

### 2. 指令存储器模块 (Instruction ROM)
*   **功能描述**：用于存储编译好的 32 位定长 RISC-V 机器指令。
*   **编址映射（重点）**：
    *   RISC-V 架构采用**字节编址**（每个地址对应 1 字节），32 位指令占 4 字节，故合法指令地址的最低两位 `PC[1:0]` 始终为 `00`。
    *   Logisim 的 ROM 组件是**字编址**（每个地址对应 4 字节）。
    *   **地址转换关系**：为了将 32 位的 PC 地址映射到 16 位地址线的 ROM，需要丢弃 PC 的最低两位 `PC[1:0]`（舍弃字节偏移）以及高 14 位 `PC[31:18]`（超出 256KB 寻址范围），仅提取 **`PC[17:2]`** 作为 ROM 的 16 位输入地址。

### 3. 下地址生成逻辑 (Next PC Logic)
*   **功能描述**：根据控制信号 `NPCASrc` 和 `NPCBSrc` 的配置，实时计算 `Next PC`，以支持顺序执行、条件分支跳转（b-type）和无条件跳转（jal, jalr）。
*   **逻辑计算公式**：
    *   当 `NPCASrc = 0, NPCBSrc = 0` 时（顺序执行）：$\text{Next PC} = \text{PC} + 4$
    *   当 `NPCASrc = 0, NPCBSrc = 1` 时（JAL 或 Branch 成功）：$\text{Next PC} = \text{PC} + \text{Imm}$
    *   当 `NPCASrc = 1, NPCBSrc = 1` 时（JALR 指令）：$\text{Next PC} = \text{BusA(Rs1)} + \text{Imm}$

## 三、 接口信号与引脚定义

| 信号名称 | 方向 | 位宽 | 物理属性 | 功能定义 |
| :--- | :--- | :--- | :--- | :--- |
| **CLK** | 输入 | 1 | 时钟信号 | 系统主时钟，**上升沿**触发 PC 的更新。 |
| **Reset** | 输入 | 1 | 复位控制 | **高电平有效**。为 1 时，在 CLK 到来时将 `Initial Address` 装入 PC。 |
| **Initial Address**| 输入 | 32 | 数据总线 | 系统的程序入口地址（例如 `0x00000400`）。 |
| **Imm** | 输入 | 32 | 数据总线 | 由译码部件（IDU）扩展输出的 32 位立即数。 |
| **BusA** | 输入 | 32 | 数据总线 | 源寄存器 1（rs1）读取出的 32 位数据（用于 jalr）。 |
| **NPCASrc** | 输入 | 1 | 控制信号 | 选择下地址加法器输入端 A：`0` 代表当前 PC，`1` 代表 BusA。 |
| **NPCBSrc** | 输入 | 1 | 控制信号 | 选择下地址加法器输入端 B：`0` 代表常量 4，`1` 代表立即数 Imm。 |
| **PC** | 输出 | 32 | 数据总线 | 当前正在执行的 32 位指令物理地址。 |
| **IR** | 输出 | 32 | 数据总线 | 从 Instruction ROM 中取出的当前 32 位机器指令。 |

## 四、 数据及控制信号传输通道流程分析

### 1. 顺序取指通路（PC = PC + 4）
*   **控制状态**：`NPCASrc = 0`，`NPCBSrc = 0`
*   **数据流向**：
    1.  `PC` 寄存器的当前值同时输出至 `ROM` 的地址端（经过截断为 `PC[17:2]`）和加法器 A 输入选通 MUX。
    2.  加法器 B 选通 MUX 选中常量 `4`。
    3.  专用加法器计算出 $\text{PC} + 4$，送回 PC 寄存器的输入端。
    4.  时钟上升沿到来，PC 载入新值 $\text{PC} + 4$。

### 2. 无条件/有条件相对跳转通路（PC = PC + Imm）
*   **控制状态**：`NPCASrc = 0`，`NPCBSrc = 1`
*   **数据流向**：
    1.  加法器 A 端 MUX 选中当前 `PC`。
    2.  加法器 B 端 MUX 选通当前译码产生的立即数 `Imm`。
    3.  专用加法器计算出 $\text{PC} + \text{Imm}$ 并送入 PC 寄存器输入端，完成跳转地址的预备。

### 3. 寄存器间接跳转通路（PC = BusA + Imm）
*   **控制状态**：`NPCASrc = 1`，`NPCBSrc = 1`
*   **数据流向**：
    1.  加法器 A 端 MUX 切换至 `1`，读取寄存器堆输出的 `BusA`。
    2.  加法器 B 端 MUX 切换至 `1`，读取立即数 `Imm`。
    3.  专用加法器计算出 $\text{BusA} + \text{Imm}$（即 `jalr` 的目标地址），在下个上升沿锁存至 PC。

# 5.4取操作数部件 IDU 实验
## 整体方案设计

## 一、 顶层设计模块图与数据通路

IDU 部件主要由**指令拆解分线器（Splitter）**、**立即数扩展器（ImmGen）**、**32×32 寄存器堆（RegFile）**以及**操作数选通多路选择器（MUX）**四大部分构成。其顶层数据通路与控制逻辑如下：

```text
                     +-------------------------------------------------+
                     |                 取操作数部件 IDU                |
                     |                                                 |
                     |             +-----------------+                 |
                     |      +----> | 立即数扩展器    |                 |
                     |      |      |   (ImmGen)      | --------+ (Imm) |
                     |      |      +-----------------+         |       |
                     |      |               ^                  |       |
                     |      |               | (ExtOp)          |       |
                     |   [31:0]             |                  |       |
     IR (指令输入) ----+------+---------------------------------+       |
                     |      |                                  |       |
                     |      |      +-----------------+         |       |
                     |      +----> | 寄存器堆 RegFile|         |       |
                     |    rs1,rs2  | - 32 Word x 32  |         |       |
                     |    rd, RegWr| - x0 始终为 0    |         |       |
                     |             +-----------------+         |       |
                     |                |           |            |       |
                     |                v           v            v       |
                     |              BusA        BusB          Imm      |
                     +-------------------------------------------------+
                                      |           |            |
                                      v           v            |
                                   +----+      +----+          |
                     PC (当前地址) -|1  0|-     |1  2| <--------+
                                   |MUX |      |MUX |
                                   +----+      +----+
                                     |           |
                                     v           v
                                   DataA       DataB (送往 ALU 输入端)
```

---

## 二、 子模块详细描述与技术实现

### 1. 指令拆解与分线器模块 (Instruction Splitter)
*   **功能描述**：由于 RV32I 指令集格式极其规整，分线器通过静态物理连线将 32 位指令 `IR` 直接拆分为以下字段：
    *   `opcode` = `IR[6:0]` (操作码)
    *   `rd` = `IR[11:7]` (目标寄存器索引)
    *   `funct3` = `IR[14:12]` (功能码 3)
    *   `rs1` = `IR[19:15]` (源寄存器 1 索引)
    *   `rs2` = `IR[24:20]` (源寄存器 2 索引)
    *   `funct7` = `IR[31:25]` (功能码 7)

### 2. 立即数扩展器模块 (ImmGen)
*   **功能描述**：根据控制信号 `ExtOp[2:0]`，将指令中分散的立即数片段拼接并**符号扩展（Sign-Extension）**或**零扩展**至 32 位。
*   **扩展格式规则（支持 5 种指令类型）**：
    *   **I-Type** (`ExtOp = 000`)：`immI = { {20{IR[31]}}, IR[31:20] }`
    *   **U-Type** (`ExtOp = 001`)：`immU = { IR[31:12], 12'b0 }`
    *   **S-Type** (`ExtOp = 010`)：`immS = { {20{IR[31]}}, IR[31:25], IR[11:7] }`
    *   **B-Type** (`ExtOp = 011`)：`immB = { {19{IR[31]}}, IR[31], IR[7], IR[30:25], IR[11:8], 1'b0 }` *(注：最低位强制补 0)*
    *   **J-Type** (`ExtOp = 100`)：`immJ = { {11{IR[31]}}, IR[31], IR[19:12], IR[20], IR[30:21], 1'b0 }` *(注：最低位强制补 0)*

### 3. 寄存器堆模块 (RegFile)
*   **功能描述**：包含 32 个 32 位通用寄存器（`x0` ~ `x31`）。支持双端口同步读、单端口有条件写。
*   **关键约束**：
    *   **`x0` 寄存器硬线恒置 0**：无论写入什么数据，从 `x0` 读取的结果必须恒为 `0x00000000`。
    *   **时序规范**：读取操作为纯组合逻辑（给出 `rs1`/`rs2` 立即输出 `BusA`/`BusB`）；写入操作为时序逻辑，在时钟 **上升沿** 且写使能 `RegWr = 1` 时，将 `BusW` 写入 `rd` 指定的寄存器。

### 4. 操作数多路选择模块 (Operand MUX)
*   **功能描述**：根据 `ALUASrc` 和 `ALUBSrc` 选择最终送往 ALU 的两个数据源。
    *   **ALU A 输入端选择 (ALUASrc MUX)**：
        *   `0`：选通 `BusA`（寄存器数据）。
        *   `1`：选通 `PC`（当前指令物理地址，用于 `auipc` 或 `jal`）。
    *   **ALU B 输入端选择 (ALUBSrc MUX)**：
        *   `00`：选通 `BusB`（寄存器数据）。
        *   `01`：选通 **常量 4**（用于顺序执行或链路计算）。
        *   `10`：选通 32 位扩展立即数 **`Imm`**。

## 四、 核心数据流控制流程分析

### 1. 寄存器-寄存器型指令（R-Type, 如 `add`）
*   **控制状态**：`ExtOp = x`, `ALUASrc = 0`, `ALUBSrc = 00`, `RegWr = 1`（在下一个上升沿写回结果）。
*   **数据流向**：
    1.  `IR` 的 `rs1` 和 `rs2` 传入 RegFile。
    2.  RegFile 组合逻辑输出 `BusA` 和 `BusB`。
    3.  两个 MUX 分别选通 `BusA`（`ALUASrc=0`）和 `BusB`（`ALUBSrc=00`）。
    4.  输出 `DataA = BusA`, `DataB = BusB` 直达 ALU。

### 2. 寄存器-立即数型指令（I-Type, 如 `addi`）
*   **控制状态**：`ExtOp = 000` (I-Type 扩展), `ALUASrc = 0`, `ALUBSrc = 10` (选通立即数)。
*   **数据流向**：
    1.  `IR` 传入 RegFile 读取 `rs1`，输出至 `BusA`；MUX 选通 `BusA` 输出 `DataA`。
    2.  `IR[31:20]` 传入 `ImmGen`，根据 `ExtOp = 000` 进行 32 位符号扩展，输出 `Imm`。
    3.  `ALUBSrc` 选通 `2`（即 `10` 进制），将扩展后的 `Imm` 作为 `DataB` 输出。

# 5.5数据通路实验
个人理解：本实验存在三个时序逻辑部件（接入clk）：IFU（取指令）/IDU（decode）/DataRAM（存储），在时钟上升沿到来时，这三个部件同时更新状态、采取行动，这是一个CPU周期的开始。上升沿过去后电平保持稳定，各路信号行进。IFU依据输入计算出下一个指令地址并存入PC；IDU对依据旧PC取出的指令进行解码吐出各种控制信息，依据ALUASrc/ALUBSrc控制MUX，得到DataA/DataB并送入ALU，ALU在ALUctr的控制下进行相应的运算，计算出result,并将result[0]和zero信号送到BandJ部件，加上BandJ信号依据编码表输出NPCASrc/NPCBSrc（送往IFU）；同时DataRAM将旧的result[0:17]作为地址进行访问，由MemOp决定读取与写入方式，在MemtoReg信号控制下得到BusW，决定将什么数据送往IDU的寄存器堆。所有过程在下一个时钟上升沿到来前完成并达到电平稳定（建立时间、保持时间）  。DataRAM模拟的是内存(memory)，IDU内部的RegFile是寄存器堆。
## 整体方案设计

单周期 CPU（Single-Cycle CPU）的数据通路设计旨在实现“在一根时钟周期内，完成单条指令的所有执行阶段（取指、译码、执行、访存、写回）”。该设计不仅要求各部件功能正确，更要求在物理连线上实现严格的数据同步与正确的控制路由。

## 子模块详细描述与技术实现

### 1. 跳转控制部件 (BandJ)
*   **功能描述**：它是 CPU 跳转分支判定的大脑。在执行 `B-Type` 或 `J-Type` 指令时，该部件接收译码器输出的 3 位 `BandJ` 控制编码，并结合 ALU 实时输出的 `Zero` 标志位和运算结果最低位 `Result[0]`（在小于比较指令中代表 $A < B$ 是否成立），通过组合逻辑计算生成控制 IFU 进行跳转的两个关键信号：`NPCASrc` 和 `NPCBSrc`。
*   **逻辑表达式推导**：
    *   根据表 5.6 的功能定义，通过卡诺图或代数化简可得逻辑表达式：
        $$\text{NPCASrc} = \text{BandJ}[1] \cdot \overline{\text{BandJ}[2]}$$
        $$\text{NPCBSrc} = \text{BandJ}[0] + \text{BandJ}[2] \cdot \text{Branch\_Cond}$$
    *   其中，分支条件成立信号 $\text{Branch\_Cond}$ 的组合逻辑表达式为：
        $$\text{Branch\_Cond} = (\text{BandJ}[2:0] == 4 \cdot \text{Zero}) + (\text{BandJ}[2:0] == 5 \cdot \overline{\text{Zero}}) + (\text{BandJ}[2:0] == 6 \cdot \text{Result}[0]) + (\text{BandJ}[2:0] == 7 \cdot (\text{Zero} \cup \overline{\text{Result}[0]}))$$

### 2. 取指部件 (IFU)
*   **功能描述**：为了实现程序可视化和顶层连线的规整，将原指令存储器（Instruction ROM）移出 IFU 子电路，单独放置在顶层数据通路中。
*   **电路修改点**：
    *   **移除内部 ROM**：删除原 `lab5.3` 电路中的 ROM 存储器，直接引出当前 `PC` 寄存器的 32 位物理输出端。
    *   **删除旧 IR 端口**：在 IFU 外观上删除 32 位输入端口 `IR`，仅保留下地址计算通路、时钟 `CLK` 输入和复位 `Reset` 输入。
    *   **地址截断**：在顶层引出 `PC` 信号后，利用分线器（Splitter）提取 `PC[17:2]`（16 位宽）接入顶层单独部署的 256KB ROM。

### 3. 取操作数与译码部件 (IDU)
*   **功能描述**：负责解析 ROM 输出的 32 位机器码 `IR`，并在内部 `RegFile` 中读取源寄存器数据。
*   **总线选通机制**：
    *   根据 `ALUASrc`，选通输出 `DataA`（寄存器数据 `BusA` 或当前 `PC`）。
    *   根据 `ALUBSrc`，选通输出 `DataB`（寄存器数据 `BusB`、常量 `4` 或扩展后的 32 位立即数 `Imm`）。

### 4. 算术逻辑运算部件 (32-bit ALU)
*   **功能描述**：接收 `DataA` 和 `DataB` 作为输入操作数，根据 4 位控制信号 `ALUctr` 执行算术、逻辑或比较运算。
*   **输出反馈**：将 32 位主输出连接到数据存储器地址端和写回多路选择器；同时将 `Zero` 和 `Result[0]` 信号实时反馈给 `BandJ` 部件，用于跳转判定。

### 5. 数据存储器 (DataRAM)
*   **功能描述**：用于执行 `lw` / `sw` 等访存指令。
*   **容量与编址**：容量为 256KB，数据位宽 32 位，但采用**字节编址**（物理寻址需要 18 位地址线）。
*   **地址线连接（重点）**：由于采用字节编址，且本次实验假设数据访问均满足 **自然对齐（Natural Alignment）**，故将 ALU 计算出的 32 位内存地址 `ALU_Out` 的低 18 位 `ALU_Out[17:0]` 直接连接到 DataRAM 的地址输入端 `A`。

## 四、 控制信号与数据传输通道分析

### 1. 寄存器写回通路选择（MemtoReg MUX）
在数据存储器的输出端和 ALU 输出端之间，需要一个 **2 选 1 多路选择器** 来决定写回到 `RegFile` 的数据源：
*   当 `MemtoReg = 0` 时：写回数据 `BusW` 来源于 **ALU 的计算结果**（适用于所有算术、逻辑、移位和比较指令）。
*   当 `MemtoReg = 1` 时：写回数据 `BusW` 来源于 **数据存储器 DataRAM 的输出端口**（专门用于 `lw` / `lhu` / `lbu` 等加载指令）。

### 2. 条件跳转指令数据流流程（以 `blt` 为例）
*   **控制信号状态**：`ALUctr = 2` (小于比较), `BandJ = 6` (blt)
*   **数据流动通路**：
    1.  `IFU` 读出 `PC` 地址，`ROM` 输出 `blt` 机器码至 `IR`。
    2.  `IDU` 译码，从 `RegFile` 中读出两个寄存器的值作为 `BusA` 和 `BusB`，分别作为 `DataA` 和 `DataB` 传入 `ALU`。
    3.  `ALU` 执行小于比较，将比较结果（0 或 1）通过最高位/最低位反馈至 `Result[0]`。
    4.  `BandJ` 部件接收 `BandJ=6` 和 `Result[0]`。
        *   若 `Result[0] = 1`（A < B 成立）：`BandJ` 产生 `NPCBSrc = 1`，通知 `IFU` 载入跳转目标地址 $\text{PC} + \text{Imm}$。
        *   若 `Result[0] = 0`：`BandJ` 产生 `NPCBSrc = 0`，通知 `IFU` 顺序执行 $\text{PC} + 4$。

### 1. 信号漏连（如 `Din`、`BusA` 悬空）
*   **现象**：测评中出现 `xxxxxxxx` 或者是最后 `jalr` 跳转时 PC 归零（变为 4）。
*   **本质**：在复杂的层次化设计（Hierarchical Design）中，底层模块（如 IDU、RegFile）虽然内部逻辑 100% 正确，但由于顶层连线（Top-Level Routing）漏掉了一根线，导致信号无法在模块间“合流”。
*   **工程教训**：在设计复杂 SoC 时，必须建立严格的**接口检查表（Interface Checklist）**。每一个子模块的 Out 端口必须明确流向，每一个 In 端口必须有确定的驱动源（Driver），绝对不能留有悬空（Floating）引脚。

### 2. 截错总线位宽（如 `Result0` 截取了 Bit 31 错当 Bit 0）
*   **现象**：`blt` 小于比较跳转指令在测评中永远判定为“不成立”，无法跳转。
*   **本质**：将 `Result[31]`（符号位）误当作了 `Result[0]`（比较结果位）。在 32 位数据通路中，总线是高度并行的，**某一位（Bit）的偏离都会导致整个控制流的彻底失效**。
*   **工程教训**：位宽截取与错位是处理器设计中最隐蔽的 Bug。在 Logisim 中使用分线器（Splitter）时，必须极其敏感地核对 **Bit Mapping（位映射）属性**。

## 1
 要在 3 个 LED 点阵（总宽度 24 像素）上滚动显示 5 个字符（总宽度 40 像素），核心在于构建 **“时间维度的滑动窗口寻址与像素级逻辑移位”**。 
 
---

多点阵多字符像素级平滑滚动显示器设计思路总结

本设计基于 **“动态滑动窗口（Sliding Window）”** 与 **“统一行扫描显存（Unified Scanline Buffer）”** 架构，攻克了在物理宽度受限的 3 个 LED 点阵（24 列像素）上，平滑、循环滚动显示 5 个离散字符（“NJUCS”，总宽度 40 像素）的硬件设计难题。

其核心设计思想可概括为三个维度：

### 一、 空间维度：多路地址并行选通与“滑动窗口”重构
由于“NJUCS”五个字符在只读存储器（ROM）中的 ASCII 码地址（字模物理空间）是不连续的，本设计摒弃了简单的地址累加，构建了**四路并行的动态地址选通网络**：
1.  **时间基准**：由一个 6 位循环计数器产生当前的像素滚动偏移。其中，低 3 位 `Pixel[2:0]` 代表字符内部的像素级偏移（0~7）；高 3 位 `Char[2:0]` 代表滑动窗口在 5 个字符间的起始索引（0~4）。
2.  **错位取模控制**：通过部署加法器和比较器构成的**“减 5 修正选择逻辑”**，在硬件上以极低的门延迟实现了：
    *   $\text{Base}_0 = \text{Char} \pmod 5$
    *   $\text{Base}_1 = (\text{Char} + 1) \pmod 5$
    *   $\text{Base}_2 = (\text{Char} + 2) \pmod 5$
    *   $\text{Base}_3 = (\text{Char} + 3) \pmod 5$
3.  **并联寻址**：4 个地址选择 MUX 同时动态选通当前窗口覆盖的 4 个字符的 ASCII 码，送入 4 列并行的 ROM 阵列中。这保证了在任何滚动瞬间，电路都能**同步、实时、无延迟地读出这 4 个字符的全部字形数据**。

---

### 二、 时间维度：矩阵级高速并行读取
为了彻底消除闪烁并实现瞬时刷新，本设计采用了**空间并行化（Spatial Parallelism）** 思想：
1.  每一列 ROM 内部通过 $16$ 级加法器链（`+1` 字节偏移），在同一个时钟周期内，同时读出单个字符的全部 $16$ 行数据。
2.  ４列并联，使得整个电路在任意瞬间，能够并行吐出 $4 \times 16 = 64$ 字节的点阵字模。
3.  对于任意行 $i$（Row 1 ~ Row 16），利用分线器将 4 个独立字符的 $8$ 位行点阵拼接为 **$32$ 位宽的单行大总线数据**：
    $$\text{Line\_32}[31:0] = \{\text{Char}_3[7:0], \text{Char}_2[7:0], \text{Char}_1[7:0], \text{Char}_0[7:0]\}$$

---

### 三、 执行维度：分布式桶形移位与固定视口截取
像素级的平滑滚动（从右向左缓缓滑入滑出）最终通过最右侧的**分布式桶形移位阵列**完成：
1.  **分布式移位**：电路右侧并排部署了 $16$ 个 32 位移位器（或等价选通选择器）。每一行 $i$ 的 32 位大总线数据 `Line_32` 送入移位器的输入端，由低位计数器 **`Pixel[2:0]`** 实时驱动移位端的移动。
2.  **视口截取（窗口化）**：随着时钟滴答，移位器内部的字形数据以 1 像素为单位向左平移。移位器的输出端固定通过分线器**截取最高的 24 位**（分别对应左、中、右 3 个 LED 点阵的同一行输入）。
3.  **视觉效果**：当 `Pixel` 满 8 溢出导致 `Char` 递增时，窗口自动切入下一个字符，物理上呈现出完美、平滑、无缝的 ASCII 字符左右像素级滚动效果。

## 2

若 PC 寄存器、寄存器堆和数据存储器写入数据的时钟信号触发边沿不一致（例如 PC 在上升沿触发，而 RegFile/DataRAM 在下降沿触发），将对 CPU 的时序收敛与逻辑正确性产生致命影响：
1.  **导致建立时间违例**：
	 单周期 CPU 的核心机制是利用一个时钟周期 $T$ 供长链组合逻辑进行信号传导与收敛。若写回器件（如 RegFile）改为下降沿触发，则留给组合逻辑传播的时间将被迫减半，变为 $T/2$。在高速运行或执行复杂指令（如带符号移位、减法比较）时，信号无法在半个周期内稳定，导致写回器件锁存住未收敛的过渡态“垃圾数据”，程序运行崩溃。
  2.  **引发保持时间违例与数据竞争**：
	时钟边沿不一致会引入严重的时钟偏斜与竞争冒险。例如，若 DataRAM 提前在下降沿锁存数据，而在下一个时钟上升沿前，PC 或译码控制信号若发生微小抖动，会导致输入端的数据在写入周期结束前提前发生跳变（无法满足保持时间限制），造成内存写入脏数据或发生严重的数据读写冲突。
3.  **结论**：
	单周期数据通路必须保持严格的同步时序设计。所有状态主控器件（PC, RegFile, DataRAM）必须严格对齐在同一种时钟边沿（统一上升沿触发更新），以确保控制流与数据流的相位一致性与物理鲁棒性，保证程序正常进行。
## 3

存在以下三种方法：
### A：软件死循环（RISC-V 官方标准做法，最轻量）

查询资料，发现在实际的 RISC-V 裸机开发或操作系统关机前，最常用的方法是在程序末尾放一条 “自循环”指令。

*   **实现方式**：
    在汇编程序结束时，加入以下指令：
    ```assembly
    loop: jal x0, loop   # 或者 beq x0, x0, loop
    ```
*   **硬件行为**：
    *   这条指令的机器码被读出后，`BandJ` 判定跳转永远成立，计算出的下地址 `NextPC` 永远等于当前 `PC` 的值。
    *   在每个时钟上升沿，PC 寄存器不断地将当前 PC 重新锁存进去。
*   **优缺点分析**：
    *   **优点**：不需要对 CPU 增加任何一个门电路的修改，100% 靠软件实现，兼容性极佳。
    *   **缺点**：**CPU 并没有真正“停机”**。时钟信号 clk 仍在跳动，ALU 和数据通路依然在进行无谓的计算。

---

### 方案 B：硬件使能封锁（PC 写使能控制，最优雅）

为了真正从硬件上让 PC “冻结”，我们可以利用 PC 寄存器的 **“写使能”** 端。

*   **实现方式**：
    1.  **定义停机指令 (HALT)**：
        在指令集里自定义一条停机指令。当译码器（IDU）解析到这条指令的 `opcode` 时，控制单元（Control Unit）输出一个控制信号 **`Halt = 1`**。
    2.  **锁定 Halt 信号**：
        为了防止 Halt 信号转瞬即逝，我们用一个 **SR锁存器** 或一个 **1位触发器** 锁存住 `Halt` 状态。一旦触发 `Halt = 1`，锁存器输出 `Halted = 1`，直到系统外部复位（Reset）。
    3.  **封锁 PC 的 `en` 端**：
        *   在之前的设计中，PC 寄存器的写使能端 `en` 是直接接常量 `1`（始终允许更新）的。
        *   现在，我们将 `PC` 的 `en` 端接上：`~Halted`）。
*   **硬件行为**：
    *   当执行到 `HALT` 指令的瞬间，`Halted` 变为了 `1`，`PC` 的使能端 `en` 瞬间变为了 `0`。
    *   此后，无论时钟信号 `clk` 如何跳动，PC 寄存器都会拒绝写入任何新数据，从而牢牢地停在 `HALT` 指令的下一条地址上。
*   **优点**：非常安全，PC 彻底冻结，且数据通路不再产生新的状态翻转，没有额外的功耗。

---

### 方案 C：硬件时钟门控（Clock Gating，最硬核）

在工业级低功耗设计中，最彻底的停机是直接把时钟掐断。

*   **实现方式**：
    *   同样通过译码产生 `Halted` 锁存信号。
    *   我们不把全局时钟 `clk` 直接连到 PC，而是将 `clk` 与 `~Halted` 通过一个与门相连，产生门控时钟 `Gated_clk` 送往 PC 寄存器：
        $$\text{Gated\_clk} = \text{clk} \cdot \overline{\text{Halted}}$$
*   **硬件行为**：
    *   一旦 `Halted = 1`，与门的输出被彻底锁定为低电平（`0`），时钟方波被无情抹平。
    *   没有了时钟沿，PC 寄存器、RegFile、DataRAM 彻底失去触发动力，CPU 进入“假死/深度休眠（Sleep）”状态，动态功耗直接降为 0。
*   **注意**：门控时钟容易产生时钟毛刺，在设计时需要确保 `Halted` 信号的翻转远离时钟沿。
