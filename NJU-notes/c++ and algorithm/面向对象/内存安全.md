# 2. 内存安全问题包括哪些？请阐述每种问题的发生场景，并使用C++代码各举一个例子。
## (1)内存泄漏
发生场景：在堆上动态分配了内存，但是在 程序运行结束之前/指向该内存的所有指针都失效之前，忘记释放该内存，导致这部分的内存无法再被使用。
示例：
```c++
void memory_leak_example() {
    int* ptr = new int(100);
    // 函数直接返回，没有调用 delete ptr; 
    // 局部变量 ptr 在栈上被销毁，而堆上的 100 无法被访问也无法被释放，导致内存泄漏。
}
```
## (2)悬空指针
发生场景：指向的堆内存已被释放，但是指针变量本身没有置空，随后程序再次尝试通过该指针读写内存。
示例
```c++
void use_after_free_example() {
    int* ptr = new int(42);
    delete ptr; // 内存已被释放，ptr 变为悬空指针
    
    std::cout << *ptr << std::endl; // 错误！访问已释放的内存，导致未定义行为（Undefined Behavior）
}
```

# (3)缓冲区溢出/越界访问
发生场景：对固定大小的数组或缓冲区进行越界读写，例如使用下标 `i >= size` 或字符串拷贝时未检查长度。
示例：
```c++
char buf[4]; strcpy(buf, "hello"); // 实际长度 6 (含 '\0')，超出 buf 容量，发生溢出
```
## (4)重复释放
发生场景：对同一块内存调用两次 `delete`（或 `free`），通常因多个指针指向同一地址且分别释放，或者代码分支逻辑混乱。
示例：
```c++
int* p = new int(7); int* q = p; delete p; delete q; // double free！p 和 q 指向同一块内存
```
## (5)野指针/未初始化内存
发生场景：指针变量在声明的时候**没有初始化**，此时它的值是随机的垃圾地址。直接对野指针进行解引用会使程序尝试访问位置的随机内存。
示例：
```c++
void wild_pointer_example() {
    int* ptr; // 未初始化，它是一个野指针
    *ptr = 10; // 错误！向一个随机地址写入 10，轻则程序当场崩溃，重则破坏其他敏感内存
}
```


# 3. 请解释什么是RAII，说明使用RAII 的优点以及C++中常见的实现方式。
### 概念
RAII(Resource Aquisition Is Initialization，资源获取即初始化)是c++的核心编程范式，核心思想为：**将资源的生命周期绑定到局部对象（栈对象）的生命周期上**。
- 资源在对象的**构造函数**中获取（如申请堆内存、打开文件等），在对象的**析构函数**中释放（如释放内存、关闭文件等）。
编译器保证局部对象离开作用域时自动调用析构函数（无论正常离开还是异常抛出）。
### 优点
- **异常安全**：即使发生了异常，析构函数也会被调用，资源不会泄漏。
- **简化代码**：无需手动配对 `new/delete` 或 `lock/unlock`，减少遗忘风险；极大减少了繁琐的 try-catch 或各种退出判断分支里的资源清理代码。
- 1. **资源管理自动化**：无需手动编写释放代码，避免重复释放、悬空指针等错误。
### C++中常见的实现方式
- **智能指针**：`std::unique_ptr`、`std::shared_ptr`、`std::weak_ptr`
- **标准容器**：`std::vector`、`std::string`：内部动态内存自动管理
- **锁**：`std::lock_guard`、`std::unique_lock`：构造时加锁，析构时解锁。
- **文件流**：std::ifstream 和 std::ofstream（构造时打开文件，析构时自动关闭文件）。
- 自定义RAII类：将资源封装在类中，构造函数获取，析构函数释放。

### 2. 内存安全问题包括哪些？请阐述每种问题的发生场景，并使用C++代码各举一个例子。

常见的内存安全问题主要包括以下 5 种：

#### (1) 内存泄漏 (Memory Leak)
*   **发生场景**：在堆上动态分配了内存，但在程序运行结束前、或是指向该内存的所有指针都失效前，**忘记释放该内存**，导致这部分内存无法再被使用。
*   **C++ 示例**：
    ```cpp
    void memory_leak_example() {
        int* ptr = new int(100);
        // 函数直接返回，没有调用 delete ptr; 
        // 局部变量 ptr 在栈上被销毁，而堆上的 100 无法被访问也无法被释放，导致内存泄漏。
    }
    ```

#### (2) 悬空指针 / 释放后使用 (Dangling Pointer / Use-After-Free)
*   **发生场景**：指向的堆内存已经被释放（`delete`），但指针变量本身没有置空，随后程序**再次尝试通过该指针读写内存**。
*   **C++ 示例**：
    ```cpp
    void use_after_free_example() {
        int* ptr = new int(42);
        delete ptr; // 内存已被释放，ptr 变为悬空指针
        
        std::cout << *ptr << std::endl; // 错误！访问已释放的内存，导致未定义行为（Undefined Behavior）
    }
    ```

#### (3) 重复释放 (Double Free)
*   **发生场景**：同一块动态分配的堆内存被**执行了两次或多次释放（`delete`）**，通常发生在多个指针指向同一块资源，或者代码分支逻辑混乱时。这会导致堆管理器的数据结构损坏，引发程序崩溃甚至安全漏洞。
*   **C++ 示例**：
    ```cpp
    void double_free_example() {
        int* p1 = new int(10);
        int* p2 = p1; // p1 和 p2 指向同一块内存
        
        delete p1;
        delete p2; // 错误！重复释放了同一块内存，程序会崩溃
    }
    ```

#### (4) 缓冲区溢出 / 越界访问 (Buffer Overflow / Out-of-bounds Access)
*   **发生场景**：程序向数组（栈、堆或全局数组）中写入或读取了**超出其分配边界**的数据。
*   **C++ 示例**：
    ```cpp
    void buffer_overflow_example() {
        int arr[5] = {1, 2, 3, 4, 5};
        // 合法索引是 0 ~ 4，arr[5] 是越界访问
        arr[5] = 99; // 错误！向数组外的栈内存写入数据，可能覆盖其他局部变量，导致安全漏洞
    }
    ```

#### (5) 野指针 / 未初始化指针 (Wild Pointer / Uninitialized Pointer)
*   **发生场景**：指针变量在声明时**没有初始化**，此时它的值是随机的垃圾地址。直接对野指针进行解引用会使程序尝试访问未知的随机内存。
*   **C++ 示例**：
    ```cpp
    void wild_pointer_example() {
        int* ptr; // 未初始化，它是一个野指针
        *ptr = 10; // 错误！向一个随机地址写入 10，轻则程序当场崩溃，重则破坏其他敏感内存
    }
    ```

### 3. 请解释什么是RAII，说明使用RAII 的优点以及C++中常见的实现方式。

#### (1) 什么是 RAII？
**RAII (Resource Acquisition Is Initialization，资源获取即初始化)** 是 C++ 的核心编程范式。
它的核心思想是：**将资源的生命周期绑定到局部对象（栈对象）的生命周期上**。
*   **获取资源**（如申请堆内存、打开文件、加锁等）发生在对象的**构造函数**中。
*   **释放资源**（如释放内存、关闭文件、解锁等）发生在对象的**析构函数**中。

因为 C++ 保证局部对象离开作用域时一定会自动调用析构函数，所以与之绑定的资源也就百分之百能够得到正确释放。

#### (2) 使用 RAII 的优点
1.  **绝无资源泄漏**：无需程序员手动编写释放代码（如 `delete`、`close`），由编译器和对象生命周期保证自动释放。
2.  **异常安全 (Exception Safety)**：如果代码在执行中抛出异常，C++ 会启动**栈展开（Stack Unwinding）**，自动调用已构造栈对象的析构函数。手动编写的 `delete` 代码可能会被异常跳过，但 RAII 绝对不会失效。
3.  **代码简洁高效**：极大减少了繁琐的 `try-catch` 或各种退出判断分支里的资源清理代码。

#### (3) C++ 中常见的 RAII 实现方式
*   **内存管理**：`std::unique_ptr` 和 `std::shared_ptr`（把裸指针封装成栈对象，析构时自动 `delete`）。
*   **多线程锁管理**：`std::lock_guard` 和 `std::unique_lock`（构造时加锁 `mutex.lock()`，析构时自动解锁 `mutex.unlock()`）。
*   **文件流管理**：`std::ifstream` 和 `std::ofstream`（构造时打开文件，析构时自动关闭文件）。
*   **标准容器**：`std::vector` 和 `std::string`（内部自动管理动态数组的开辟和释放）。

### 4. 举例说明 unique_ptr 能防护哪些内存安全问题，并给出一个不能防护的场景。

`std::unique_ptr` 是独占式智能指针，它是 RAII 思想在堆内存管理上的完美体现。

#### (1) unique_ptr 能防护哪些内存安全问题？

*   **防护 1：内存泄漏 (Memory Leak)**
    *   **原理**：`unique_ptr` 在析构时自动调用 `delete` 释放其所管理的内存。
    *   **例子**：
        ```cpp
        void safe_function() {
            auto uptr = std::make_unique<int>(100);
            if (some_error) {
                return; // 即使中途直接返回，uptr 被销毁，内存依然会被释放，不会发生泄露
            }
        } // 正常退出，内存同样被安全释放
        ```

*   **防护 2：重复释放 (Double Free)**
    *   **原理**：`unique_ptr` **禁用了拷贝构造和拷贝赋值**，只允许**移动 (Move)**。这意味着同一块内存不可能被两个 `unique_ptr` 同时管理。
    *   **例子**：
        ```cpp
        auto p1 = std::make_unique<int>(10);
        // auto p2 = p1; // 编译报错！禁止拷贝，彻底杜绝了两个智能指针同时管理、析构时 double free 的隐患。
        
        auto p3 = std::move(p1); // 允许移动所有权，此时 p1 变为空指针，只有 p3 拥有该内存
        ```

*   **防护 3：野指针 (Wild Pointer)**
    *   **原理**：`unique_ptr` 默认构造时会自动初始化为 `nullptr`，不会带有随机的垃圾地址。

#### (2) unique_ptr 不能防护的场景有哪些？

虽然 `unique_ptr` 很强大，但如果**越过智能指针的限制使用裸指针（Raw Pointer）**，它将失去防护能力。

##### 场景 A：使用 `.get()` 获取裸指针后导致的“释放后使用 (Use-After-Free)”
`unique_ptr` 允许通过 `.get()` 获取内部的裸指针以便与老旧的 C 风格接口对接，但如果把这个裸指针保存下来并在 `unique_ptr` 销毁后继续使用，就会发生 UAF。

*   **不能防护的代码示例**：
    ```cpp
    void unsafe_scenario() {
        int* raw_ptr = nullptr;
        {
            auto uptr = std::make_unique<int>(42);
            raw_ptr = uptr.get(); // 获取内部裸指针
        } // uptr 离开作用域，生命周期结束，调用 delete 销毁了堆上的 42
        
        // 此时 raw_ptr 已经变成悬空指针
        std::cout << *raw_ptr << std::endl; // 错误！Use-After-Free，unique_ptr 无法阻止你通过 raw_ptr 越权访问
    }
    ```

##### 场景 B：用同一个裸指针初始化多个 `unique_ptr` 导致的“重复释放 (Double Free)”
如果你强行绕过智能指针的安全分配函数（如 `make_unique`），用同一个裸指针初始化了两个不同的 `unique_ptr`，就会诱发 Double Free。

*   **不能防护的代码示例**：
    ```cpp
    void unsafe_init_scenario() {
        int* raw = new int(10);
        std::unique_ptr<int> p1(raw);
        std::unique_ptr<int> p2(raw); // 危险！两个独立的 unique_ptr 都在管理同一个 raw 指针
    } // p2 和 p1 析构时都会尝试 delete raw，引发 Double Free 崩溃
    ```

##### 场景 C：动态数组的“越界访问 (Buffer Overflow)”
虽然 `unique_ptr` 支持数组特化版本 `std::unique_ptr<int[]>`，但它**不提供越界边界检查**。
*   **不能防护的代码示例**：
    ```cpp
    void arr_overflow() {
        auto arr = std::make_unique<int[]>(5);
        arr[10] = 100; // 错误！越界写内存，unique_ptr 无法在运行时或编译期阻止此行为
    }
    ```
