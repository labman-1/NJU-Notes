# 1. 请解释 dynamic variable 与local variable 在存储区域、生命周期、内存管理责任三个方面的区别。

| 维度     | dynamic variable                                                                                | lcoal variable                                    |
| ------ | ----------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| 存储区域   | 存储在**堆区**。堆内存是由操作系统管理的自由存储区，容量巨大，分配速度较栈区更慢。                                                     | 存储在**栈区**，栈内存由编译器自动分配和释放，地址通常由高向低增长，增长速度极快。       |
| 生命周期   | **动态生命周期**。变量从执行`new`或`malloc`分配空间时存在，直到程序员手动调用`delete`或`free`释放时销毁，其生命周期不受代码块作用域的限制。           | **自动生命周期**，仅在块内部存在，离开局部代码块时变量立即自动销毁。              |
| 内存管理责任 | 由**程序员**负责（除智能指针之外）。程序员必须确保每一次分配内存(`new`/`malloc`)都必须对应一次释放(`delete`/`free`)，管理不当容易导致内存泄漏或悬空指针。 | 由**编译器/操作系统**负责。程序员无需干预，绝对不会发生因为忘记手动释放内存而导致的内存泄露。 |
# 2. 内存安全问题包括哪些？请阐述每种问题的发生场景，并使用C++代码各举一个例子。
## (1)内存泄漏
发生场景：在堆上动态分配了内存，但是在 程序运行结束之前/指向该内存的所有指针都失效之前，忘记释放该内存，导致这部分的内存无法再被使用。
示例：
```c++
void memory_leak_example() {
    int* ptr = new int(100);
    // 函数直接返回，没有调用 delete ptr; 
    // 局部变量 ptr 在栈上被销毁，而堆上的 100 无法被访问也无法被释放，导致内存泄漏。
}
```
## (2)悬空指针
发生场景：指向的堆内存已被释放，但是指针变量本身没有置空，随后程序再次尝试通过该指针读写内存。
示例
```c++
void use_after_free_example() {
    int* ptr = new int(42);
    delete ptr; // 内存已被释放，ptr 变为悬空指针
    
    std::cout << *ptr << std::endl; // 错误！访问已释放的内存，导致未定义行为（Undefined Behavior）
}
```

# (3)缓冲区溢出/越界访问
发生场景：对固定大小的数组或缓冲区进行越界读写，例如使用下标 `i >= size` 或字符串拷贝时未检查长度。
示例：
```c++
char buf[4]; strcpy(buf, "hello"); // 实际长度 6 (含 '\0')，超出 buf 容量，发生溢出
```
## (4)重复释放
发生场景：对同一块内存调用两次 `delete`（或 `free`），通常因多个指针指向同一地址且分别释放，或者代码分支逻辑混乱。
示例：
```c++
int* p = new int(7); int* q = p; delete p; delete q; // double free！p 和 q 指向同一块内存
```
## (5)野指针/未初始化内存
发生场景：指针变量在声明的时候**没有初始化**，此时它的值是随机的垃圾地址。直接对野指针进行解引用会使程序尝试访问位置的随机内存。
示例：
```c++
void wild_pointer_example() {
    int* ptr; // 未初始化，它是一个野指针
    *ptr = 10; // 错误！向一个随机地址写入 10，轻则程序当场崩溃，重则破坏其他敏感内存
}
```


# 3. 请解释什么是RAII，说明使用RAII 的优点以及C++中常见的实现方式。
### 概念
RAII(Resource Aquisition Is Initialization，资源获取即初始化)是c++的核心编程范式，核心思想为：**将资源的生命周期绑定到局部对象（栈对象）的生命周期上**。
- 资源在对象的**构造函数**中获取（如申请堆内存、打开文件等），在对象的**析构函数**中释放（如释放内存、关闭文件等）。
编译器保证局部对象离开作用域时自动调用析构函数（无论正常离开还是异常抛出）。
### 优点
- **异常安全**：即使发生了异常，析构函数也会被调用，资源不会泄漏。
- **简化代码**：无需手动配对 `new/delete` 或 `lock/unlock`，减少遗忘风险；极大减少了繁琐的 try-catch 或各种退出判断分支里的资源清理代码。
- 1. **资源管理自动化**：无需手动编写释放代码，避免重复释放、悬空指针等错误。
### C++中常见的实现方式
- **智能指针**：`std::unique_ptr`、`std::shared_ptr`、`std::weak_ptr`
- **标准容器**：`std::vector`、`std::string`：内部动态内存自动管理
- **锁**：`std::lock_guard`、`std::unique_lock`：构造时加锁，析构时解锁。
- **文件流**：std::ifstream 和 std::ofstream（构造时打开文件，析构时自动关闭文件）。
- 自定义RAII类：将资源封装在类中，构造函数获取，析构函数释放。

# 4. 举例说明unique_ptr 能防护哪些内存安全问题，并给出一个不能防护的场景。
## (1)能防护的内存安全问题
- 内存泄漏：`unique_ptr` 在其析构时自动 `delete` 所管理对象，无需手动释放。
- 悬空指针：所有权唯一，不能拷贝。通过移动语义转移所有权后，原指针自动置 `nullptr`，避免多个指针指向已释放内存。
- 重复释放：因为不能拷贝，不存在两个 `unique_ptr` 同时指向同一块内存，杜绝了双重释放的可能。
- 野指针（一定程度上）：`unique_ptr` 必须初始化（通过 `new` 或 `make_unique`），不会出现未初始化的智能指针。
## (2)不能防护的场景
虽然 `unique_ptr` 解决了所有权方面的常见问题，**但它无法防护以下场景**（主要源于对裸指针的滥用或数组越界）：

- **使用 `get()` 获取裸指针后手动进行非法操作**  
    例如：通过 `get()` 获得内部裸指针并错误地 `delete` 它，导致未定义行为。
    
    ```cpp
    auto p = std::make_unique<int>(10);
    int* raw = p.get();
    delete raw; // UB：unique_ptr 析构时会再次 delete，造成 double free
    ```
    
- **缓冲区溢出**  
    如果 `unique_ptr` 管理的是动态数组（`std::unique_ptr<int[]>`），它不检查下标越界，访问越界仍会导致内存破坏。
    
    ```cpp
    auto arr = std::make_unique<int[]>(5);
    arr[10] = 42; // 越界写入，unique_ptr 无法阻止
    ```
    
- **移动后继续使用原裸指针**  
    若通过 `get()` 获取指针后发生移动，原裸指针成为悬空指针。
    
    ```cpp
    auto p = std::make_unique<int>(1);
    int* raw = p.get();
    auto p2 = std::move(p); // p 变为 nullptr
    *raw = 2;               // UB：raw 已成为悬空指针
    ```
    
- **与原始指针混用时忘记所有权转移**  
    例如将 `unique_ptr` 传给不接受智能指针的旧 API，API 内部可能 `delete` 该指针，导致后续 `unique_ptr` 析构时再次释放。

**总结**：`unique_ptr` 是强力工具，但并不能完全替代对内存安全的全面检查，仍需避免裸指针滥用、越界访问等错误。
