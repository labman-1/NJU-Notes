## Interface inheritance(a.k.a what)
- subclass inherits signatures, but NOT implementation.
- Allows you to generalize code in a powerful, simple way.  
### Step 1:Defining a List61B
use the keyword **interface** instead of **class** to define a List61B.
-  Idea:Interface us a soecification of what a list is able to do, not how to do it.
```java
public interface List61B<Item>{
public void addLast(Item x);
}
```
### Step 2:Implementing the List 61B Interface
-  use the keyword **implements** to tell the Java comliler that SLList and AList are hyponyms of List61B.
```java
public class AList<Blorp> implements List61B<Blorp>{
}
```
Subclass *must* override all of these methods!
## Method Overriding vs. Overloading
If a "subclass" has a method with the exact **same signature** as in the "superclass", we say the subclass **overrides** the method (only applies to inheritance relationship).
Methods with the same name but different signatures are **overloaded**.
## Implementation inheritance(a.k.a how)
-  subclass can inherit signatures AND implementation.
- Allows code-reuse: Subclasses can rely on superclasses or interfaces. Gives another dimension of control to subclass designers: Can decide whether or not to override default implementations.
use th **default** keyword to specify a method that subclasses should inherit from an **interface**.
在父类中使用default完成method的具体内容（可以使用其他method，因为它们都会被继承），这样在子类中可以直接使用此方法

## implements vs. extends
在 Java 中，implements/extends 两个关键字虽然都用于描述“继承”或“关系”，但它们的应用场景和**契约强度**完全不同。

我们可以从**生物学（继承）**、**社会学（契约）**和**代码规范**三个维度来解析。

### 1. 核心定义：血缘与合同

#### **Extends (扩展/继承) —— “血缘关系”**
*   **主体**：类与类（Class extends Class） 或 接口与接口（Interface extends Interface）。
*   **哲学**：**"Is-a" (是一个)**。比如，猫 **extends** 哺乳动物。
*   **含义**：子类不仅继承了父类的“长相”（签名），还继承了父类的“家产”（具体的代码实现）。
*   **限制**：在 Java 中，一个类只能有一个亲爹（**单继承**），防止逻辑混乱（经典的“菱形继承”问题）。

#### **Implements (实现) —— “签署合同”**
*   **主体**：类与接口（Class implements Interface）。
*   **哲学**：**"Can-do" (能做某事)**。比如，猫 **implements** 捕猎接口。
*   **含义**：类向外界承诺：“我保证我能完成这些功能，至于怎么做，由我来写”。接口里通常只有“要求”，没有具体的代码。
*   **能力**：一个类可以签署多份合同（**多实现**）。比如，一个类既可以 `implements Iterable`，也可以 `implements Serializable`。

### 2. 你刚才遇到的具体场景

在你的代码中，为什么我说 `Deque` 应该 **`extends`** `Iterable`？

```java
public interface Deque<T> extends Iterable<T> { ... }
```

1.  **同类相吸**：`Deque` 是接口，`Iterable` 也是接口。接口和接口之间是“能力的叠加”，所以用 `extends`。这意味着“双端队列”**是一种**可迭代的东西。
2.  **能力的扩充**：这相当于在说，凡是以后实现 `Deque` 的类，必须不仅能处理队列操作，还得能提供“导游”（Iterator）服务。

而当你写类的时候：
```java
public class LinkedListDeque<T> implements Deque<T> { ... }
```
这是类在向接口“宣誓”，要把接口里那些虚幻的要求（What to do）变成扎实的代码（How to do）。

### 3. 三个维度的对比总结

| 维度 | `extends` | `implements` |
| :--- | :--- | :--- |
| **关键词** | **继承 (Inheritance)** | **实现 (Implementation)** |
| **关系类型** | `类与类` 或 `接口与接口` | `类与接口` |
| **获得什么** | 获得父类的**属性**和**方法实现** | 获得接口的**方法签名**（必须自己写内容） |
| **数量限制** | **单继承** (只能有一个父类) | **多实现** (可以签署无限个接口) |
| **哲学** | 这是一个什么东西？ | 这个东西能做什么？ |

### 4. 深度哲学：为什么 Java 不允许类多继承，却允许接口多继承？

这是设计上的**权衡（Trade-off）**：

*   **类的多继承是危险的**：如果 A 类和 B 类都有一个 `eat()` 方法且实现不同，C 类同时 `extends` A 和 B，当 C 调用 `eat()` 时，它该听谁的？（这叫**致命方块问题**）。
*   **接口的多继承是安全的**：如果接口 A 和接口 B 都有一个 `eat()` 的签名，接口 C `extends` A 和 B。因为接口里没有具体的代码实现，C 只需要知道“我必须实现一个 `eat()`”就行了，不存在听谁的问题。
