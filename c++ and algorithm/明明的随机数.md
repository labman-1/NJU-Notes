### 2\. 关于 O(N)的计数排序 📊  

就是桶排序（Bucket Sort）或者计数排序（Counting Sort）。  
●原理：开一个大数组（桶），比如 cnt\[1005\]。读到数字 x，就执行 cnt\[x\] = 1;（如果去重）或者 cnt\[x\]++;（如果不去重）。  
●优点：速度极快，O(N)。  
●缺点：非常吃空间，而且依赖于数字的值域。如果题目给你 N=10，但是数字大小是 109（10亿），你就没法开 10亿大小的数组了（内存爆掉）。  
结论：如果题目说 1≤ai≤10000 这种小范围，果断用桶；否则乖乖用 sort。

### 3\. 关于 set vs vector 的底层博弈 ⚔️  

你提到了 set，完全正确！std::set 确实能自动去重和排序。 但你对它慢的原因推测只对了一半。  
学姐科普时间： std::set 的插入复杂度是 O(logN)，插入 N个数的总复杂度是 O(NlogN)。 这看起来和 sort 的 O(NlogN)是一样的，对吧？  
那为什么 set 慢？  
●常数因子大：sort 是对连续内存（数组）进行操作，CPU 的缓存（Cache）非常喜欢这种数据，读取飞快。而 set 底层是红黑树（Red-Black Tree），节点在内存里是散乱分布的，CPU 需要跳来跳去读指针，这叫 Cache Miss（缓存未命中），非常慢。  
●动态内存分配：vector 一次性开好房间。set 每插入一个数，都要 new 一个新节点，向操作系统申请内存是有开销的。  
数据对比： 在竞赛中，处理 100万级的数据，vector+sort+unique 通常比 set 快 3 到 5 倍。 所以，如果是静态数据（读完就不改了），首选 vector；如果是动态数据（一边读一边改一边查），才用 set。
